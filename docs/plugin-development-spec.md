# LiteLauncher 插件开发规范

更新时间：2026-03-29
适用版本：LiteLauncher `v1.0.12`

## 1. 目标

这份规范用于统一 LiteLauncher 插件的目录结构、命令协议、面板行为、交互规范和回归要求，确保：

1. 结构统一：目录、接口、执行入口一致。
2. 体验统一：搜索命中、状态提示、键盘行为一致。
3. 质量可控：插件可测试、可灰度、可回归。

## 2. 目录与命名

每个插件必须独立目录，不允许多个插件混写在同一文件里。

- 主进程入口：`src/main/plugins/<plugin-id>/index.ts`
- `<plugin-id>` 规则：小写 + 中划线，例如 `webtools-jwt`
- 统一在 `src/main/plugins/index.ts` 注册
- 默认是否对外可见由“可见插件白名单”决定，可在设置页按插件 ID 调整

## 3. 主进程接口约束

插件必须实现 `LauncherPlugin`：

- `id`：全局唯一
- `name`：展示名称
- `createCatalogItems()`：空输入时的入口项
- `getQueryItems(query)`：搜索时的动态结果
- `execute(optionsText, context)`：执行入口，返回 `ExecuteResult`

要求：

- 所有参数校验都在插件内部完成
- 不能把未捕获异常直接抛到上层
- 非法输入返回 `ok: false` + 明确错误文案
- 需要图标的插件，必须提供稳定图标来源

## 4. LaunchItem 规范

- `id`：稳定且唯一
- `type`：插件统一使用 `command`
- `target`：统一协议
  - `command:plugin:<plugin-id>?action=<action>&...`
- `title` / `subtitle`：可读、优先中文
- `keywords`：覆盖中文关键词 + 英文别名
- `iconPath`：必须可解析，支持本地路径或 `data:`
- 若同一目标能命中多条结果，至少一条要保留完整标题、可执行 target 和可显示图标

## 5. 命令协议

### 5.1 目标格式

```text
command:plugin:<plugin-id>?action=<action>&k=v...
```

### 5.2 动作建议

- 必须支持：`open`
- 按需支持：`parse`、`generate`、`convert`、`verify`、`review` 等

### 5.3 参数处理

- `execute` 内做完整参数校验
- 参数越界时要给用户明确反馈
- 插件不得依赖渲染层保证参数一定合法

## 6. 面板协议（Renderer）

打开插件统一走 `openPanel(panel=plugin)`：

```ts
{
  panel: "plugin",
  pluginId: string,
  title?: string,
  subtitle?: string,
  message?: string,
  data?: Record<string, unknown>
}
```

渲染层通过 `pluginId` 分发处理器，不新建独立 `BrowserWindow`。

如果插件有默认示例、默认表达式或默认参数，打开面板后应自动给出首屏结果，避免用户看到空白区。

## 7. 交互规范

- `Enter`：执行当前插件主动作
- 多行输入框内 `Enter`：换行
- 多行输入框内 `Ctrl+Enter`：执行主动作
- `Esc`：返回搜索页
- 关键结果要通过状态栏提示成功 / 失败
- 耗时动作需要加载态或“处理中”提示
- 自动联动型插件应在输入变化后自动刷新结果，但仍需保留明确主动作供 `Enter` 触发

## 8. 原生交互规范

如果插件会触发系统文件选择框、保存对话框、下载动作或其他会导致窗口失焦的原生交互，必须接入“自动隐藏暂停”链路。

要求：

- 原生交互开始前挂起自动隐藏
- 交互结束后及时恢复
- 避免窗口长期保持不自动隐藏

典型场景：

- 图片选择
- 二维码下载
- 文件保存
- 图片导出

## 9. 搜索规范

至少支持：

- 英文别名，例如 `wt-jwt`
- 中文关键词，例如“加密”“密码”“二维码”
- 必要时补拼音首字母或别名映射

匹配优先级：

- 精确 > 前缀 > 受限模糊

长串随机噪声输入不应产生成批误命中。

## 10. 图标规范

- 每个插件必须有可识别图标
- 搜索结果、目录入口、插件面板入口图标保持一致
- 本地图标文件应确保主进程可直接读取
- 如果使用静态图片，优先提供稳定本地资源文件

## 11. 错误与状态处理

- 空输入：给出可理解提示，不应直接报错
- 参数错误：返回明确反馈，不让界面静默失败
- 外部依赖失败：说明原因，同时保持面板可继续操作
- 未捕获异常：统一写入错误日志链路
- 系统能力调用结果至少区分成功 / 取消 / 失败

## 12. 测试与验收清单

每个插件至少通过以下检查：

1. 空输入下可在插件分区显示。
2. 关键词搜索可命中。
3. `Enter` 可执行主动作。
4. `Esc` 可返回搜索页。
5. 状态提示正确。
6. 图标显示正常。
7. 小屏窗口无关键布局错乱。
8. `pnpm run build` 通过。

## 13. WebTools 系列额外要求

针对 `webtools-*` 插件：

1. 必须保留 LiteLauncher 原生插件结构，不能直接把旧页面整包搬回。
2. 目标是功能与原 `webTools` 等价，不以“能打开”替代“已完成”。
3. 每个工具独立目录，禁止多个工具堆在一个文件里。
4. 公共逻辑要沉淀到共享层，避免 19 个插件重复造轮子。
5. 只有通过主流程、键盘、小屏和回归检查后，才进入默认可见列表。
